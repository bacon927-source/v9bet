Sign-up UX 決議記錄

日期：2026-07-15（07-16 補充 Verification Flow）· 形式：線上逐項確認（截圖標註 → 修改 → 驗證）· 產出以 signup-flow-wireframe.html 為單一事實來源，其他頁面同步

一、註冊（Sign Up）

1註冊方式 = 電話（預設分頁）／Email（分頁切換）＋密碼

  • 表單只有三件事：📱 Phone number｜✉️ Email 分頁 → 帳號欄位 → Password（👁 顯示切換）→ 21+/Terms 勾選。
  • 電話分頁輸入數字 → 彈出國碼選擇，只有 🇹🇭 泰國 +66 / 🇻🇳 越南 +84（依語系/geo-IP 預設）。在電話分頁貼上 email 會自動切到 Email 分頁。
  • 推薦碼移出註冊頁（原欄位由密碼取代）。

歷程：原本國碼+電話+推薦碼 → 改單一欄位自動判斷 → 最終定案加回明確的雙分頁（單一欄位邏輯保留在偵測行為）。

2語言／幣別／推薦碼延後 — 不影響「成為客人」

帳號在第一頁＋OTP（或社群授權）完成當下即建立；第一頁越短完成率越高。三項延後各有配套：

延後項目風險配套（已寫入 spec）
推薦碼歸因掉失邀請連結 ?ref= 自動帶碼；7 天內可在優惠頁補填
幣別不選不能入金Setup Step 1 必填＋入金當下強制補選（擋入金、不擋註冊）
語言看不懂介面語系/geo-IP 自動預設＋頁首常駐切換器

要盯的指標：「完成第一頁 → 完成 Setup Step 1」轉換率。

3第三方註冊 = Google / LINE / X 圓形 icon 列

  • 只放 icon（不做整條按鈕）；Telegram 入口移除（僅保留 TH/intl 的 OTP 頻道用途）；Apple / Facebook 不提供（政策）。icon 顯示依市場自動隱藏。
  • 社群註冊全程免 OTP——provider 已驗證身份；流程：授權（無站內 redirect 畫面）→ 最小化 Complete Setup（幣別+21+，無推薦碼）→ 直接顯示 「You're now a V9 customer」🎉 歡迎頁。
  • 手機 OTP 延後到提款／領優惠時才要求（電話仍是 KYC anchor）。

二、驗證（OTP / Verification）

4OTP 頻道 = Zalo ZNS / SMS / LINE / WhatsApp / Email

  • 預設依市場：VN = Zalo ZNS、TH = LINE；SMS 為通用備援；失敗 fallback = WhatsApp / Email。
  • 移除：Telegram bot code、語音報碼（Voice call）。

channel(zalo_zns|sms|line|whatsapp|email) — 註冊 OTP 與 step-up 驗證共用同一組清單

5獨立 Verification Flow 頁面（07-16 新增，依 Unified OTP docx 對齊）

規格來源：《Unified OTP Authentication Flow Enhancement》docx。核心：系統分開維護 Email / 電話的驗證狀態（Verified / Not Verified），四種狀態決定 OTP 方式——無驗證→依動作而定、僅 Email→Email OTP、僅電話→Phone OTP、雙驗證→會員自選。

  • 適用 7 種動作：首次綁定、加另一種聯絡方式、改密碼、換綁 Email/電話、提款、領優惠、忘記密碼。
  • 同類型規則：改 Email 第一步 OTP 只能發到舊 Email（電話 OTP 不能代替）；改電話同理。新聯絡方式兩段 OTP 都通過才生效。
  • 無驗證聯絡方式：改密碼／提款 = 不允許；領優惠／首次綁定 = 現場輸入並驗證新聯絡方式後繼續。
  • 忘記密碼：不得在驗證成功前洩漏帳號是否存在；用於復原的聯絡方式不自動變 Verified（待業務確認）。
  • OTP 通則：有效 5–10 分鐘；系統需定義過期/錯誤上限/重發間隔/重發上限/失敗後動作/重用政策/信任期；OTP 綁定 會員+聯絡方式+動作+session；稽核日誌不存明碼。

5b獨立 Login Flow 頁面（07-16 新增）

登入也拆成獨立 flow，與 sign-up 同格式：Login 主畫面（社群置頂＋Last used、One-time code｜Password 分頁）→ 一次性驗證碼登入（驗證過的聯絡方式＋頻道）→ 生物辨識/Passkey（WebAuthn＋降級）→ 忘記密碼（走 Unified OTP 流程）→ 鎖定與風險 step-up＋session 規則。

5cOTP 設計稿對齊（07-16，dark theme mockups）

收到完整 OTP 元件視覺稿＋六個接入點，verification flow 已照此重寫。重點決議：

  • OTP 模式只有 Email / SMS（Select Verification Mode 單選）；Zalo ZNS / LINE / WhatsApp 遞送列為 phase 2。
  • 元件參數：6 位數碼、重發倒數 32 秒、錯誤顯示「Invalid OTP entered」（紅框紅鈕、倒數續跑）、第 3 次重發後冷卻升級 03:00、聯絡方式一律遮罩、支援 iOS 簡訊自動帶碼。
  • 無驗證聯絡方式 → 不再擋任何動作：所有入口（含改密碼、提款）都改為「現場輸入並驗證新聯絡方式，該驗證即當作 OTP」——覆蓋 docx 的 Not allowed。
  • 改密碼 = 2FA：移除舊密碼欄位，OTP 即身份驗證；改完強制重新登入。
  • 提款 OTP 觸發條件：收款帳戶（銀行帳號或 USDT 地址）首次使用時才要求 OTP。
  • 領優惠：先補 Personal Details（Full Name 需與銀行帳戶名完全一致＋生日）→ OTP → 繼續領取。
  • 忘記密碼：選模式（Email/SMS）→ 輸入註冊聯絡方式 → OTP 通過後才顯示 Account ID → 設新密碼（含強度計）；適用 Active / Dormant 帳號。
  • Profile 聯絡方式：首次輸入後電話移除自行編輯（未驗證狀態仍可 Edit）；換綁需先驗當前同類型聯絡方式，收不到碼 → 找客服；「Verified Email may be used as Account ID」。

verification-flow-wireframe.html 重寫為 8 畫面（規則 → Select Mode → First-time Verify → OTP 狀態 → 結果 → 換綁 → 忘記密碼 → 接入點）；login-flow 忘記密碼同步

四、幣別與 Setup

7帳戶幣別只有 VND / THB —— USDT 不是獨立幣別

USDT 是包在 VND / THB 帳戶內的充提通道（即時匯率換算）；幣別選擇畫面移除 USDT 選項、兩幣別標註「USDT supported」。入金/出金的 Bank/USDT 切換照舊。

8Setup 設定頁 = 三步驟＋頂部進度條

步驟內容可否跳過
Step 1 · Required帳戶幣別、Profile + bank/USDT不可 pass
Step 2 · SecurityFace ID / biometric / passkey、備援 Email、密碼/PIN（註冊已設，此處為修改）可 pass
Step 3 · Preferences通知權限、主題/語言可 pass

教學（Interactive tutorial）與 App 下載移出清單 —— Setup 完成後在下一頁（Tutorial & App 單一頁）一起呈現，之後才是 Welcome + Offers。
